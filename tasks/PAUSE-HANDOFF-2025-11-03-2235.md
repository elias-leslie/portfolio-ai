## Root Cause Analysis

**The Perfect Storm (4 issues):**

1. **Connection leak**: `celery_app.control.inspect()` objects never closed
   - Every call leaked a network connection
   - 47 CLOSE_WAIT connections accumulated

2. **Duplicate workers**: shutdown.sh didn't kill Celery beat
   - Multiple restarts accumulated 34+ worker processes and 2 beat processes
   - Each inspect() call had to query all workers → 40+ seconds

3. **Slow operation in hot path**: TASK-0028 added `check_celery_worker()` with `inspect.stats()` to `/health` endpoint
   - `/health` called every 2-5 seconds by frontend
   - Each call took 40+ seconds with 35 worker processes
   - Requests queued up faster than they could be served

4. **Aggressive polling**: Frontend `useSystemStatus` hook polled `/health` every 5 seconds
   - SSE failed due to overload → fell back to polling
   - With 2s retries and 2 retry attempts, created cascading load

**Fix Strategy**:
- Close inspect() objects (fixes leak)
- Kill Celery beat in shutdown (fixes accumulation)
- Skip inspect() in `/health` endpoint (fixes hot path)
- Increase polling to 30s (reduces load)

## Key Decisions

**Architecture:**
- Separated fast health checks (`/health` with `skip_slow_checks=True`) from detailed status (`/api/status` without skip)
- Status Dashboard still gets full Celery worker details via `/api/status` endpoints
- Health endpoint optimized for frequent polling by monitoring tools

**Patterns:**
- Added `skip_inspect: bool = False` parameter to `check_celery_worker()`
- Added `skip_slow_checks: bool = False` parameter to `get_all_service_statuses()`
- All `inspect()` calls now use try/finally with timeout=2.0

**Issues Discovered:**
- TASK-0028/0029/0030 introduced performance regression
- Need to be more careful about operations in frequently-called endpoints
- shutdown.sh was incomplete (missing Celery beat)

**Band-Aids Avoided:**
- Did NOT add caching layers
- Did NOT add rate limiting
- Did NOT disable features
- Fixed ROOT CAUSE instead of symptoms
